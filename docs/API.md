# API Documentation Summary

Full interactive documentation (Swagger UI) is auto-generated by FastAPI and available at `http://127.0.0.1:8000/docs` whenever the backend is running â€” this document is a summary for quick reference, not a replacement for it.

Base URL: `{API_BASE}/api/v1` (default local: `http://127.0.0.1:8000/api/v1`)

Auth: Bearer JWT in the `Authorization` header, obtained from `POST /auth/login`. Send as `Authorization: Bearer <token>`.

## Endpoints

| Method | Path | Auth | Summary |
|---|---|---|---|
| POST | `/auth/register` | **None (public)** | Register a new user. See `SECURITY.md` â€” this needs restricting before production. |
| POST | `/auth/login` | None (must be public) | Exchange username/password for a bearer token |
| POST | `/patients` | ASHA only | Register a new patient |
| GET | `/patients` | Any authenticated user | Search patients by village/name (default limit 20, no pagination) |
| GET | `/patients/{patient_id}` | Any authenticated user | Get a patient by ID |
| POST | `/screenings` | ASHA only | Submit a diabetes screening â€” runs the full Verify -> Risk -> Referral pipeline |
| GET | `/screenings/{screening_id}` | Any authenticated user | Get a screening result by ID |
| GET | `/screenings/patient/{patient_id}` | Any authenticated user | Get a patient's screening history |
| GET | `/referrals` | Any authenticated user | List referrals, optionally filtered by `status` |
| PATCH | `/referrals/{referral_id}` | PHC_OFFICER only | Update a referral's status (PENDING -> REFERRED -> COMPLETED) |
| GET | `/dashboard` | DISTRICT_OFFICER only | Aggregate district dashboard summary |
| GET | `/health` | None | Liveness check |
| GET | `/` | None | Service info |

**"Any authenticated user"** means all three roles can call it â€” there's no further row-level restriction (e.g. an ASHA can view any patient/screening/referral, not just ones in their own village). See `SECURITY.md` for why this is flagged as a known limitation, not a bug.

## Key Request/Response Shapes

### `POST /screenings` (the core endpoint)

Request (`ScreeningCreate`):
```json
{
  "patient_id": 1,
  "blood_glucose_mg_dl": 220.0,
  "bmi": 29.5,
  "family_history_diabetes": true,
  "physical_activity_level": "LOW",
  "symptoms": ["fatigue", "frequent_urination"],
  "village_at_screening": "Wadgaon",
  "latitude": null,
  "longitude": null
}
```

Response (`ScreeningResult`):
```json
{
  "id": 42,
  "patient_id": 1,
  "screened_at": "2026-07-12T10:30:00",
  "blood_glucose_mg_dl": 220.0,
  "bmi": 29.5,
  "confidence_score": 91.2,
  "confidence_status": "HIGH",
  "confidence_breakdown": {
    "clinical_consistency_score": 96.0,
    "historical_consistency_score": 70.0,
    "behaviour_consistency_score": 92.0,
    "geographic_consistency_score": 89.0
  },
  "confidence_reasons": ["Clinical consistency: High. ...", "..."],
  "risk_score": 84.1,
  "risk_level": "HIGH",
  "referral_status": "PENDING",
  "recommendation": "Referral generated â€” refer to PHC for confirmation and treatment."
}
```

`risk_score`, `risk_level`, and `referral_status` are `null` unless `confidence_status == "HIGH"` â€” this is the confidence gate, visible directly in the response shape.

### Symptom values

Must be from the backend's recognized set (`app/ai/synthetic_data.py`'s `SYMPTOM_POOL`, mirrored in the frontend at `frontend/src/lib/symptoms.ts`): `fatigue`, `frequent_urination`, `excessive_thirst`, `blurred_vision`, `slow_healing_wounds`. Other strings are accepted (no validation rejects them) but won't be recognized by Arogent Risk's feature vector.

## Error Responses

Standard FastAPI/Pydantic validation errors (422) for malformed requests, plus explicit handling for:

| Status | When |
|---|---|
| 401 | Missing/invalid/expired token |
| 403 | Valid token, wrong role for this endpoint |
| 404 | Referenced patient/screening/referral doesn't exist |
| 409 | Username already taken (register) |
| 422 | Request body fails Pydantic validation |

## Rate Limiting

**Not implemented.** No endpoint has rate limiting. This is a real gap for a public deployment (especially `/auth/login`, which has no brute-force protection) â€” see `SECURITY.md`.
